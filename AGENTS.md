# Инструкции для ИИ-помощника: Proxy Checker

## Проект и домен

Приложение для поиска работоспособных серверов Proxy.

### Основные сущности
- ProxyItem (fetcher.ProxyItem): "Сырые" данные, полученные из источника. Содержит базовую информацию (Host, Port, Type, Country, RTT). Не содержит результатов проверки.

- CheckResult (services.Result): Результат единичной проверки прокси. Содержит тайминги (TCP, HTTP), HTTP-статус, флаги поддержки протоколов и ошибку (если есть).

- ProxyItemFull (services.ProxyItemFull): Объединение ProxyItem и CheckResult. Это финальная бизнес-сущность валидного прокси.

- Config (config.Config): Глобальные настройки приложения, загружаемые из TOML. Могут быть переопределены флагами CLI.

### Ограничения

- Масштабирование: Максимальное количество concurrent-воркеров ограничено числом 512 (аппаратное ограничение во избежание исчерпания файловых дескрипторов и портов).

- Отмена операций (Context): При отмене контекста (например, кнопка "Прервать" в GUI) пайплайн должен прекратить диспатч новых задач мгновенно. Текущие горутины-чекеры обязаны корректно завершаться по контексту, даже при работе с SOCKS5 (где есть свои таймауты на уровне Dial).

- Ошибки чекеров не фатальны: Ошибка проверки одного прокси (таймаут, отказ) не прерывает пайплайн. Ошибка считается фатальной только если упал сам источник (Fetcher) или инициализация (например, чтение GeoIP БД).

- Протоколы: SOCKS4 принципиально не поддерживает HTTP/2. Попытка принудительной проверки HTTP/2 через SOCKS4 должна сразу возвращать ошибку на уровне бизнес-логики, не доходя до сети.

## Архитектура
- Чистая архитектура: разделение на доменные модели (common), бизнес-логику (services, fetcher), сетевое взаимодействие (proxies), GUI (gui) и CLI (cli)
- Внедрение зависимостей через интерфейсы, без глобальных переменных
- Архитектура строго следует принципу единой ответственности (SRP) на уровне интерфейсов:
    fetcher.Fetcher: Отвечает ТОЛЬКО за парсинг и получение сырых ProxyItem из внешних источников. Не имеет права выполнять сетевые проверки доступности прокси.

    services.Checker (функции CheckProxy / CheckBatch): Отвечает ТОЛЬКО за сетевую валидацию. Принимает на вход ProxyItem и возвращает CheckResult. Не знает о том, откуда пришел прокси (из GUI, CLI или файла).

    common.GeoIPResolver: Отвечает ТОЛЬКО за маппинг IP-адреса в строку (название страны). Абстрагирует работу с .mmdb файлами от остальной логики. Слой GUI/CLI передает его в fetcher.Settings, если источнику нужен локальный резолвинг стран (например, TheSpeedX).

    gui.CacheInterface: Отвечает ТОЛЬКО за персистентность результатов между сессиями. Не знает о сетевых проверках.

    Правило ИИ: При рефакторинге или добавлении фич категорически запрещается смешивать ответственности этих слоев. Например, нельзя добавить проверку HTTP статуса внутрь метода Fetch() интерфейса Fetcher.

- Используем common/i18n/*json вместо языковых констант в коде
- Ошибки оборачиваются через fmt.Errorf с %w
- Если мы что-то используем в сравнении более 2х раз, то это надо выносить в константы проекта.
- Если код можно переиспользовать - переиспользуй.
- Не применяй короткие названия для переменных. Названия должны раскрывать смысл сущности.
- Не используй deprecated методы, классы и т.п
- Не игнорировать обработку ошибок
- Запрещено возвращать кортеж вида (nil, nil), если функция сигнатурой предполагает возврат ошибки (например, (T, error)). Если произошла ошибка, второй аргумент должен содержать её описание; если ошибки нет — первый аргумент не должен быть nil без веской причины (например, пустой слайс предпочтительнее nil).
- Предлагать решения по возможности на версиях библоитек не ниже, чем указны в go.mod файле
- Не забывай в тестах инициализировать i18n, елси проверяется текст, который зависит от языка. Можно выбрать язык тут.
- Ресиверы в функциях должны быть указателями.
- Вместо возвращения списка объектов из методов - возвращаем список указателей, в функции так же стараемя передавать список указателей.
- Всегда проверять возвращаемые функциями ошибки
- Строки перевода не должны содержать служебные символы вида %d, %w, %s и т.д, а так же без символов перевода строк (\n, \r)
- Вся изменяемая state-логика GUI должна быть сосредоточена в одной структуре (сейчас это AppGUI) и синхронизироваться через fyne.Do() при обращении из горутин.
- Не создавать дополнительные глобальные переменные состояния в пакете gui.
- Вся работа с sync.WaitGroup, каналами jobs/results и пулами воркеров должна находиться исключительно в пакете services.
- GUI и CLI только вызывают готовые методы пайплайна (например, services.RunPipeline) и получают готовый слайс ProxyItemFull.
- GUI имеет право вызывать бизнес-логику в фоновой горутине (go func() { ... }()). Однако любое изменение виджетов Fyne (текст лейблов, прогресс-бар, данные таблицы) изнутри этой горутины обязательно должно оборачиваться в fyne.Do() или fyne.DoAndWait() для синхронизации с главным потоком UI.
- Преобразование доменных моделей (ProxyItemFull) в модели отображения (например, gui.ProxyItemWrapper для таблицы) должно происходить в слое GUI, но это должен быть простой for цикл с копированием полей. Никакой фильтрации или сортировки результатов в GUI не должно быть — сортировку по latency делает слой services.
- Запрещено предлагать патчи в виде "вставь эти 2-3 строки в функцию". ИИ обязан предоставлять код целой функции (или целого метода/структурного блока). Это жесткое правило для защиты от потери контекста: ранних возвратов (early return), замыканий, областей видимости переменных и побочных эффектов. Если функция получается слишком большой для демонстрации — это сигнал к тому, что её сначала нужно рефакторить (разбить на более мелкие).
- Ограничение размера функций: Максимальный размер функции (или метода) — 100 строк. Это жесткое правило для соблюдения SRP и упрощения тестирования. Если функция превышает порог, её нужно декомпозировать: выделить блоки логики в приватные функции с говорящими названиями. Категорически запрещено создавать "функции-обертки" (wrapper-методы), единственная цель которых — обойти лимит строк (например, вынести одно присваивание в отдельную функцию). Разбиение должно иметь реальный бизнес-смысл.
- Целостность данных при рефакторинге: При изменении типа возвращаемого значения (или аргумента) интерфейса, ИИ обязан проверить и обновить ВСЕ точки вызова этого метода во всем проекте. Если интерфейс начинает возвращать доменную модель (например, services.ProxyItemFull), а не UI-модель (например, gui.ProxyItemWrapper), слой-вызыватель (GUI или CLI) обязан самостоятельно выполнять маппинг (копирование полей). Запрещено "проливать" доменные модели напрямую в слой отображения без конвертации.
- Глубокий анализ для линтеров: При добавлении тегов (например, json, xml, db) для удовлетворения требований статического анализа (linters), ИИ обязан анализировать всю иерархию структуры. Встроенные (embedded) анонимные поля требуют указания тегов на уровне их встраивания (например, ProxyItem \json:",inline"``). ИИ не имеет права заявлять "проблема решена", пока не проследован весь граф зависимостей структуры до базовых типов.
- Типобезопасность в GUI (Fyne): Категорически запрещено использовать binding.UntypedList и binding.Untyped для хранения структур данных, тип которых известен на этапе компиляции (например, списки моделей). Использование interface{} и type assertions в колбэках таблиц считается плохой практикой (code smell). Вместо этого следует хранить данные в типизированном срезе внутри структуры GUI (например, []*ProxyItemWrapper) и вызывать table.Refresh() после изменения данных. Это обеспечивает строгую типизацию, улучшает читаемость и производительность.
- Изоляция адаптеров (Ports & Adapters): Пакеты, отвечающие за взаимодействие с внешним миром (например, fetcher, cache, proxies), не должны вкладываться внутрь пакета бизнес-логики services. Они должны находиться на одном уровне (internal/fetcher, internal/cache), чтобы подчеркивать их роль плагинов (Input/Output Ports), а не части ядра приложения. Зависимости должны направляться от services к интерфейсам адаптеров, а не наоборот (инверсия зависимостей)
- Форматирование ошибок: Используй errors.New("text") для статичных ошибок без переменных. Используй fmt.Errorf("context: %w", err) только если есть необходимость в форматировании (подстановка переменных, обертывание нижележащей ошибки).
- Контексты в тестах: Всегда используй t.Context() в качестве базового (parent) контекста в тестах вместо context.Background(). Это предотвращает утечки горутин при падении тестов. Вызов cancel() у дочернего контекста (созданного через context.WithCancel(t.Context())) безопасен и не отменяет сам t.Context().
- Чтение потоков в память: Для получения строки или слайса байт из io.Reader (например, при перехвате os.Stdout в тестах через os.Pipe) всегда используй io.ReadAll(reader). Не пытайся использовать методы вроде ReadFrom у strings.Builder, так как он не реализует интерфейс io.ReaderFrom.
- Значимые ошибки (Zero-Value Rule): Категорически запрещено использовать `_` для игнорирования ошибок, если нулевое значение (zero value) результата сломает бизнес-логику, приведет к панике или неоправданно изменит конфигурацию.
  *Обязательные кейсы для обработки (return или log + return):*
  - Парсинг данных из GUI/CLI (`strconv.Atoi`, `time.ParseDuration`): При ошибке парсинга дефолтный `0` не должен попадать в Config (`Workers=0` вызовет панику WaitGroup, `Timeout=0s` мгновенно отменит все контексты).
  - Типоутверждения (type assertions) в колбэках GUI: Всегда использовать паттерн `val, ok := obj.(*Type)`. Прямое приведение `val := obj.(*Type)` без проверки `ok` — это гарантированный `panic` при малейшем сбое фреймворка.
  - Инициализация критичных зависимостей (GeoIP, Config).
  *Исключения (где `_` допустим и идиоматичен):*
  - Запись в `io.Discard` (например, `_, _ = io.CopyN(io.Discard, resp.Body, 512)`).
  - Запись в тестовые моки (например, `_, _ = w.Write(...)` в `httptest`).
  - Освобождение ресурсов при закрытии (например, `defer zap.S().Sync()` или `conn.SetDeadline(time.Time{})`).
  - Парсинг жестко захардкоженных констант (например, `url.Parse("http://...")`)
- Локальный state UI-экранов (Scope Rule): Виджеты, состояние которых нужно читать или изменять в колбэках родительской функции (например, чтение текста из Entry при нажатии кнопки "Сохранить"), должны быть объявлены в области видимости этой родительской функции. Запрещено создавать такие виджеты внутри вспомогательных методов-хелперов (например, createSomeBox) и пытаться обращаться к ним из родительской функции — это вызовет ошибку компиляции (out of scope). Если хелпер создает сложный виджет, он должен возвращать его вместе с контейнером (как множественные возвращаемые значения). При этом временный state экранов (введенный текст, чекбоксы) категорически запрещено выносить в поля глобальной структуры AppGUI — он должен оставаться локальной переменной функции показа экрана (например, showSettingsScreen).
- Рефакторинг и декомпозиция (Scope Check): При разделении одной функции на несколько (вынос логики в хелперы), ИИ обязан явно проверить, что все переменные, использовавшиеся в старом блоке кода (например, экземпляры структур, логгеры, конфигурации), доступны в новой функции. Если переменная была локальной, она должна быть либо передана как аргумент, либо инициализирована заново внутри новой функции. Категорически запрещено "терять" инициализацию зависимостей при переносе кода.
- Пиши комментарии в коде и тексты тестовых в assetrt-функциях на английском языке.

## Стандарты
- Линтер: golangci-lint с конфигом в .golangci.yml
- Тесты: go test, testify для assertions
- Код-ревью: обязательная проверка error handling, goroutine safety, работы с context, названия переменных должны отражать суть, используем языковые файлы вместо хардкода
- Не применяй короткие названия для переменных. Названия должны раскрывать смысл сущности. 
  Исключение: имена ресиверов (receivers) в методах структур. Ресиверы должны быть короткими (1-3 символа), соответствовать типу (например, `c` для *Client, `m` для *Manager) и следовать общепринятым стандартам Go, чтобы избежать многословия в цепочках вызовов.
